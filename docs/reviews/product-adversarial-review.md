# 娃屋世界 2：产品对抗式审查

审查日期：2026-07-01

## 结论

核心闭环已可演示：严格入世 → MBTI 分身 → 双角色同行 → 分身建议 → 玩家确认 → 训练/真实标签打卡 → 虚拟物品入柜 → 同行回顾。全量页面用于展示产品结构，非核心模块按 PRD 约定保持浅交互。

## 已关闭问题

| 严重度 | 质疑 | 证据 | 处理 |
| --- | --- | --- | --- |
| P0 | 访客能否绕过创建分身 | `/map` 与受保护 API 的无会话 E2E | 页面重定向 `/onboarding`，API 返回 401 |
| P0 | 训练数据会不会冒充真实到访 | 训练同行、建议、打卡、储物柜全链路 | 所有训练状态显示“非真实打卡/训练所得”，真实打卡要求 120 米 |
| P1 | AI 分身是否只是头像标签 | 16 MBTI 行为偏好测试与地图观察排序 | MBTI 影响地点排序、语气与主动建议，外观可独立替换 |
| P1 | 玩家与分身是否被伪造成两个 GPS | 地图实现与 DOM 验收 | 玩家使用位置光点；分身只使用同一点位的视觉像素偏移 |
| P1 | “娃屋”是否偏离为装修玩法 | `/house` 文案与分类柜 | 明确定义为虚拟物品储物柜，展示来源与分类 |
| P1 | 用户真实移动时地图是否还停在初始点 | `AmapSurface` 根因审查与 E2E 地图入口 | 从一次性定位改为“开启定位跟随”，使用持续定位更新用户光点、分身位置和地图中心 |
| P1 | 地图是否缺少可拾取目标导致同行空转 | 地图碎片刷新逻辑与 `/api/map/shards/claim` | 地图默认和真实定位后都会刷新 5 个可拾取记忆碎片，领取后更新余额 |
| P1 | 娃屋四类物品是否仍然像普通列表 | `/house` 四类储物展示窗 | 新增摆件柜、衣装柜、纪念柜、明信片夹四个可点击展示窗口，显示数量、说明和代表藏品 |
| P1 | 模型失败是否导致演示中断 | 同行建议与创造台浏览器旅程 | 明确显示降级来源，返回同人格/清单内预置结果 |
| P2 | 物品结算后娃屋是否读到旧状态 | E2E 首次失败 | 关闭状态敏感链接预取，进入娃屋时读取最新存档 |

## 保留限制

- 路线、挑战、明信片、账户恢复和数据导出属于全量功能展示，当前仅提供浅层演示。
- 真实步数采集和后台持续定位未在 Web Demo 中模拟；训练步数明确标注，正式 App 需使用原生运动权限。
- Web Demo 的持续定位只在页面打开且用户点击“开启定位跟随”后运行；后台定位、轨迹历史和系统级步数需要原生 App 权限。
- 真实到场打卡规则已实现，但成都以外评审应使用训练模式。
